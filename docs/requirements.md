# コーヒー焙煎管理アプリ 要求仕様書

> Rev 2.1 / 2026-05-02

---

## 1. 概要

自宅でコーヒーを焙煎する個人ユーザーが焙煎記録を管理・分析するための Web アプリケーション。焙煎後の振り返りを主目的とし、ログ帳としての使いやすさと蓄積データの分析・可視化を両立する。

| 項目 | 内容 |
|------|------|
| 対象ユーザー | 自宅焙煎を行う個人（1 ユーザー） |
| 利用形態 | ブラウザ（ローカル動作）・ログイン不要 |
| 主目的 | 焙煎後の記録・振り返り・過去ログ比較 |
| データ保存 | IndexedDB（全データ。Tauri 移行時に SQLite へ一本化する見通しのため、マスターデータも IndexedDB に集約。ADR-0008 参照） |
| バックアップ | CSV エクスポート・インポート |

## 2. 機能スコープ

### 2.1 v1 実装スコープ

- 焙煎ログ CRUD（作成・閲覧・編集・削除）
- 生豆管理（在庫・購入情報・輸入時期）
- 焙煎機マスター管理
- 焙煎度ラベル自由登録（名前・色）
- フレーバータグ自由登録（名前・色）
- テイスティング入力（タグ＋6 軸評価）
- 外気温・湿度・天気 自動取得（Open-Meteo）
- 室内気温 手入力・外気温手動上書き
- 重量減少率 リアルタイム計算表示
- 前回ログとの差分サマリー
- ベストレシピ管理（豆ごと）
- ログ一覧 フィルター＆ソート
- 分析グラフ（折れ線・レーダーチャート）
- CSV エクスポート・インポート（固定フォーマット）
- 生豆在庫 自動デクリメント＋手動操作

### 2.2 将来拡張（v2 以降）

- 焙煎中タイマー＋ハゼ検知（録音・音声解析）
- AI 焙煎アドバイス（Claude API 活用）
- テイスティング評価軸のカスタマイズ（軸名の変更・追加）

## 5. 機能仕様詳細

### 5.1 外気温・湿度・天気の自動取得

設定画面で位置情報を取得し、緯度・経度を AppSettings に保存する。以降の焙煎記録作成時は保存済み座標を使用し、Open-Meteo API から現在の気温・湿度・天気コードを自動取得する。

| 項目 | 仕様 |
|------|------|
| 利用 API | Open-Meteo（無料・APIキー不要） |
| 取得タイミング | 焙煎記録作成フォームを開いたとき |
| 取得項目 | 外気温（°C）・相対湿度（%）・WMO 天気コード |
| 手動上書き | 外気温は手動で上書き可能。上書き時に `tempSource = "manual"` を記録 |
| 位置変更 | 設定画面から手動で座標を再取得・変更可能 |
| 取得失敗時 | エラーモーダルは表示しない。フィールドを空欄のまま手動入力に委ねる（オフライン・権限拒否・未設定を含む） |
| 天気表示 | WMO コードを絵文字にマッピングして表示（☀️🌧️❄️ 等）。`lib/weatherEmoji.ts` にマッピングテーブルを定義 |

### 5.2 重量減少率

焙煎記録フォームで `weightBeforeG`・`weightAfterG` を入力すると、減少率をリアルタイムで計算・表示する。数値は保存せず、表示時・分析時に都度計算する。

```
weightLossRate = (1 − weightAfterG / weightBeforeG) × 100  [%]
```

### 5.3 焙煎度ラベル・フレーバータグ

| 項目 | 仕様 |
|------|------|
| 登録内容 | 名前（任意文字列）・HEX カラーコード |
| 登録上限 | なし（実用的には 10 件程度を想定） |
| デフォルト | 初期起動時に推奨セットをプリセット（§4.4 参照）。ユーザーが自由に変更可 |
| フレーバータグ選択 | 焙煎記録フォームでタグを複数選択。アクティブ時はタグ背景を色付きで表示 |
| 焙煎度選択 | ラジオボタン形式。カラードットで視覚的に区別 |

### 5.4 テイスティング評価

フレーバータグ（複数選択）と 6 軸の 5 段階評価を組み合わせる。

| 評価軸 | 意味 |
|--------|------|
| Sweetness | 甘さ |
| Acidity | 酸味 |
| Body | ボディ（重さ・厚み） |
| Bitterness | 苦味 |
| Aftertaste | 余韻の心地よさ |
| Cleanliness | 雑味のなさ。焙煎ミス（焦げ・生焼け）の検出指標 |

- 各評価軸・総合スコアの入力は**星アイコン（★）**で統一する
- `cleanliness ≤ 2` のログはログ一覧カードに警告色（赤系）を付与し、焙煎ミスを一目で識別できるようにする

### 5.5 ベストレシピ管理

豆ごとに 1 件のベストログを指定できる。`Bean.bestLogId` に対象ログの id を保存する。`overallScore` が付いている場合は最高スコアのログを自動でベスト候補として提示するオプションも設ける。

### 5.6 前回との差分サマリー

ログ詳細画面に、同じ豆の直前ログと以下の項目を比較した差分パネルを常時表示する。保存直後のモーダル表示は行わない。

- 1 ハゼ・2 ハゼのタイミング差（秒）
- 重量減少率の差（%ポイント）
- 総合評価スコアの差
- 外気温・室内気温の差

### 5.7 生豆在庫管理

| 項目 | 仕様 |
|------|------|
| 在庫登録 | `Bean.stockG` に購入時の重量を入力 |
| 自動デクリメント | 焙煎ログ**新規作成時のみ** `weightBeforeG`（焙煎前重量）を `stockG` から自動減算 |
| 編集・削除時 | 在庫は変更しない。実際の豆の使用状況と記録が一致しない場合は手動で調整する |
| 手動操作 | 設定または豆詳細画面から `stockG` を任意に加減算可能。購入時の補充・焙煎外の消費・数値修正に使用 |
| 在庫不足警告 | `stockG` がマイナスになる場合は保存前に警告を表示（保存は可能） |
| 購入履歴 | 購入日・購入店・製品名・輸入時期を記録。経過月数を豆詳細画面で表示 |

### 5.8 ログ一覧 フィルター＆ソート

すべてクライアントサイドで処理する。

| 種別 | 対応項目 |
|------|----------|
| フィルター | 豆の種類・焙煎度・焙煎機・総合スコア・日付範囲 |
| ソート | 焙煎日（新しい順 / 古い順）・総合スコア・重量減少率 |

**ログ一覧カード表示項目：**

```
[焙煎度カラードット] 豆の名称              [★★★★☆]
焙煎日 · 焙煎機名
重量減少率 XX.X%  |  [天気絵文字] XX°C
[⚠️ cleanliness ≤ 2 の場合のみ警告アイコン]
```

### 5.9 分析グラフ

| グラフ種別 | 内容 | 操作仕様 |
|------------|------|---------|
| 折れ線グラフ | 豆ごとの重量減少率・ハゼタイミングの推移（横軸: 焙煎回数） | 豆セレクターで1種類を選択して表示 |
| レーダーチャート | テイスティング 6 軸の比較 | デフォルトはベストレシピログを表示。追加ログをログ一覧から選択して重ね表示可 |

### 5.10 CSV エクスポート・インポート

固定フォーマット方式。エクスポートした CSV をそのままインポートする前提とし、列マッピング UI は設けない。

| 項目 | 仕様 |
|------|------|
| エクスポート | 全焙煎ログを CSV 形式でダウンロード（Blob + download リンク） |
| インポート | CSV ファイルをアップロード後、ヘッダー行でフォーマットを自動判定して取り込む |
| 重複処理 | id が重複する場合は上書き・スキップ・手動確認のいずれかをインポート時に選択 |
| 文字コード | UTF-8（BOM 付き推奨。Excel での文字化け防止） |

### 5.11 時間入力 UI

`roastDurationSec`・`firstCrackSec`・`secondCrackSec` の入力には **`react-mobile-picker`** によるドラムロールピッカーを使用する。

| 項目 | 仕様 |
|------|------|
| 表示形式 | MM:SS（例: `07:30`）。内部では秒数に変換して保存 |
| 操作 | スマホ: 上下スワイプで回転。デスクトップ: マウスドラッグ |
| 共通コンポーネント | 3 フィールドすべてで同一コンポーネントを使い回す |

### 5.12 画面遷移ルール

| 操作 | 遷移先 |
|------|--------|
| ログ作成・編集フォームで保存 | 保存したログの詳細画面 |
| ログ削除 | ログ一覧 |
| 豆作成・編集フォームで保存 | 保存した豆の詳細画面 |
| 豆削除 | 豆一覧 |

## 6. 想定焙煎機

本アプリの主たる想定焙煎機は weroast HOME ROASTER（ボルテックス熱風式）とする。記録フォームのデフォルト値・入力範囲はこの機器の仕様に合わせて設定する。

| 項目 | 仕様 |
|------|------|
| 機種名 | weroast HOME ROASTER |
| 方式 | ボルテックス熱風式（独自技術） |
| 最大焙煎量 | 70 g / 1 回 |
| 焙煎時間範囲 | 3〜25 分（10 秒単位で設定） |
| 冷却 | 焙煎後に自動冷却（3〜6 分） |
| 最大温度 | 230°C |

### 6.1 仕様への影響

| 項目 | 対応方針 |
|------|----------|
| 焙煎時間入力 | デフォルト範囲を 3〜25 分に設定。10 秒単位のステップ入力を推奨 |
| 将来のハゼ検知 | HOME ROASTER は 1 ハゼ・2 ハゼの音が収音可能。録音解析による自動検知の実現可能性あり |
| 焙煎機マスター | 初期プリセットとして「weroast HOME ROASTER」を登録済み状態で提供 |

## 7. 技術方針

v1 は PWA（Progressive Web App）として構築する。将来的にネイティブ機能（ハゼ検知・ファイルシステムアクセス等）が必要になった場合、Tauri へのラッピングを想定し、その移行コストを最小化する設計方針をとる。

### 7.1 v1 フェーズ：PWA

| 制約条件 | 内容 |
|----------|------|
| 動作方式 | 完全ローカル動作。バックエンドサーバー不要 |
| クラウド依存 | なし。外気温取得（Open-Meteo）のみ外部 API 使用、それ以外はオフライン動作 |
| 対応デバイス | スマートフォン優先（iOS Safari / Android Chrome）。デスクトップも使用可能 |
| UI 方針 | スマホ縦持ちでの片手操作を想定したレイアウト。タップターゲットは十分な大きさを確保 |
| データ保存 | IndexedDB をメインストレージとして使用。Service Worker でオフライン対応 |
| 配布方式 | GitHub Pages で静的ファイルとして配信。ブラウザからそのまま利用可能。ホーム画面への追加も可 |

### 7.2 採用ライブラリ

| 役割 | ライブラリ |
|------|-----------|
| フレームワーク | React + TypeScript + Vite（vite-plugin-pwa） |
| ルーター | TanStack Router |
| データフェッチ・キャッシュ | TanStack Query |
| DB アクセス | Dexie.js（IndexedDB ラッパー） |
| フォーム | React Hook Form + Zod |
| スタイリング | Tailwind CSS + shadcn/ui |
| グラフ | Nivo（`@nivo/line` + `@nivo/radar`） |
| 時間入力 | react-mobile-picker |
| テーマ | Tailwind CSS `darkMode: 'media'`（システム設定追従） |

### 7.3 v2 フェーズ候補：Tauri ラッピング

以下の機能が必要になった時点で Tauri 移行を検討する。フロントエンドの React/TS コードはそのまま流用できる。

| 移行トリガー | Tauri で得られるメリット |
|---|---|
| ハゼ検知（録音・音声解析） | マイクアクセスの制限緩和。Rust バックエンドで高精度な音声処理が可能 |
| CSV ファイルの直接読み書き | ファイルシステムへの直接アクセス。Blob/download の回避策が不要になる |
| データの永続化強化 | SQLite を Rust バックエンドで管理。ブラウザデータ削除によるデータ消失リスクがなくなる |
| AI 焙煎アドバイス（ローカル LLM） | LM Studio 等のローカル LLM との連携がより容易になる |

Tauri 移行後、フロントエンドは TanStack Query + `invoke()` によるローカル IPC 通信に切り替わる。

### 7.4 移行コスト最小化のための設計指針

- データアクセスは `repositories/` 層に集約し、IndexedDB ↔ SQLite の差し替えを UI に影響させない
- ファイル操作（CSV 出力等）はサービス層に集約し、Blob/download と Tauri FS API の切り替えを 1 箇所で完結させる
- マイク録音などのネイティブ API 呼び出しは機能フラグで分岐し、PWA 環境でも graceful degradation できるようにする

### 7.5 ディレクトリ構成

```
src/
├── db/              # Dexie インスタンス・スキーマ定義
├── repositories/    # DB アクセス抽象化（Tauri 移行時にここだけ差し替え）
│   ├── beanRepository.ts
│   └── roastLogRepository.ts
├── hooks/           # TanStack Query フック（useRoastLogs, useBeans 等）
├── schemas/         # Zod スキーマ（型定義の唯一の真実の源）
├── components/      # 汎用 UI コンポーネント
├── routes/          # TanStack Router のルート定義・画面コンポーネント
│   ├── logs/
│   ├── beans/
│   ├── analysis/
│   └── settings/
├── lib/             # ユーティリティ（CSV・気象取得・計算ロジック・weatherEmoji 等）
```

## 8. 非機能要件

| 項目 | 要件 |
|------|------|
| 動作環境 | iOS Safari / Android Chrome 最新版を必須対応。デスクトップブラウザも動作すること |
| ネットワーク | 外気温・湿度・天気取得時のみオンライン必須。それ以外はオフライン動作 |
| レスポンシブ | スマートフォン縦持ち（375 px〜）を基準に設計。タップターゲットは最小 44 px |
| データ保持 | ブラウザの IndexedDB に保存。ブラウザデータ削除で消失するため CSV バックアップを推奨 |
| パフォーマンス | ログ 1000 件時点でも一覧表示・フィルターが 1 秒以内に完了すること |
| テーマ | システムの `prefers-color-scheme` に追従したダーク／ライトモード自動切替 |
